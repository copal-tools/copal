## 1. The standard work loop

For any non-trivial change, follow this sequence. Each step has a default tool.

```
explore → plan → implement → test → docs → commit → handoff?
```

| Step | Default tool | Notes |
|---|---|---|
| **explore** | Explore subagent (parallel, ≤3) | Map the area before editing. Skip only for tiny isolated changes. |
| **plan** | Plan mode → plan file under `~/.claude/plans/` | Required for anything cross-package or touching infra. Optional for one-file fixes. |
| **implement** | Edit / Write | Match existing style. No speculative abstractions. |
| **test** | `/copal-test` | Runs the right `uv` invocation per package. Add `--integration` for live-server tests. |
| **docs** | `/copal-doc-check` | Invokes the `copal-doc-curator` subagent to detect CLAUDE.md drift. |
| **commit** | `git commit` (conventional) | No `Co-Authored-By` or "Generated with Claude" trailers. PreToolUse hook enforces this. |
| **handoff** | `/copal-handoff` (optional) | If leaving work mid-stream, write `.claude/handoff.md` for the next session. |

Skip steps only when there is a reason to. "Too small to test" is rarely true.

## 2. Phase open / close ceremonies

The project is organised in numbered phases (see status table in [CLAUDE.md](./CLAUDE.md)). Open and close them ritualistically so context survives between sessions.

### Phase open — `/copal-phase-open <num> <slug>`

1. Add `## Phase N — <name>` row to the status table in umbrella [CLAUDE.md](./CLAUDE.md), status `🚧 In progress`.
2. Append a goals block to [MERGE_PLAN.md](./MERGE_PLAN.md) (or its successor `ROADMAP.md` once Phase 5 closes).
3. Update `project_status.md` memory with current phase + branch.
4. Cut a working branch `phase-N-<slug>`.

### Phase close — `/copal-phase-close`

1. Flip the phase row in umbrella CLAUDE.md to `✅ Complete`.
2. Add a `**Phase N — <theme> (<date>).**` "Shipped" bullet list mirroring the Phase 6 style.
3. Move "Tracked follow-ups" entries that are done; keep open ones at the top.
4. Update relevant per-package CLAUDE.md (gotchas, schema refs, new modules).
5. Update `project_status.md` memory.
6. Run `/copal-doc-check` for a final drift sweep.

## 3. Cross-package contract sync rule

The CopalVX ↔ CopalPM coupling is **subprocess-only** and **non-fatal** (see umbrella CLAUDE.md "Independence"). When changes touch either side of the contract:

**Trigger files:**
- `copalvx/client/copal_core/pm_hooks.py`
- `copalpm/src/copalpm/copalvx_api.py`
- Any new `copalpm <subcommand>` added to `copalpm/src/copalpm/cli.py` that pm_hooks calls
- Any new `copalvx <subcommand>` added that copalvx_api.py calls

**Required actions:**
1. Run `/copal-cross-package` (invokes the `copal-cross-package` subagent). It diffs both sides and reports drift.
2. Both packages' CLAUDE.md must reflect the change in the **same commit**.
3. Hooks must remain non-fatal — wrap in try/except, warn and continue, never raise.
4. If a CLI subcommand signature changes, version-gate the call site (the calling package must work against both old and new for one release cycle).

## 4. Session handoff pattern

When work is incomplete at end of session, generate `.claude/handoff.md` (gitignored, kept on the branch). Next session reads it on entry.

**Generated by `/copal-handoff`** from `git status` + `git log --oneline -10` + file edits this session. Should contain:

```markdown
# Handoff — <branch> — <date>

## Shipped this session
- ...

## In progress / blocked
- File: copal_core/sync.py — refactoring foo, blocked on bar
- ...

## Next concrete action
1. ...
2. ...

## Open questions for the user
- ...
```

Next session entrypoint: read `.claude/handoff.md` if it exists, then proceed.

## 5. Tier discipline (where each fact lives)

Before writing into any doc, consult the table in [CLAUDE.md](./CLAUDE.md) "Where does each fact live?". The `copal-doc-curator` subagent uses the same table. Quick reminder:

- **Code-local gotcha** → per-package CLAUDE.md
- **Cross-cutting convention** → umbrella CLAUDE.md
- **Personal preference** → user-global memory
- **Phase status** → status table + `project_status.md` memory
- **Open follow-ups** → MERGE_PLAN.md → ROADMAP.md (post-Phase-5)

Anti-patterns the curator enforces:
- No duplication of the same fact across CLAUDE.md and memory.
- Removing code = removing its CLAUDE.md mention in the same commit.
- Persistent gotchas stay in "Critical Gotchas" until the root cause is gone.

## 6. Gotcha discipline

The documented gotchas (storage contract, HKLM/24H2, encoding, data-dir migration, FK delete order, body size limit, path-traversal guard, non-fatal subprocess hooks) are **expensive to relearn**. Before merging anything significant:

```
/copal-gotcha-check
```

Invokes the `copal-gotcha-reviewer` subagent on the current diff. Output is grouped by severity (must-fix / should-fix / nit) with file:line references. Treat must-fix as blocking.

## 7. Test discipline

Before declaring a change done:

```
/copal-test                  # detects affected package(s) from diff, runs unit tests
/copal-test --integration    # adds integration tests (requires live CopalVX server)
```

Test commands by package (from umbrella [CLAUDE.md](./CLAUDE.md)):

```powershell
# CopalPM (61 tests, no external deps)
uv run --directory copalpm pytest

# CopalVX client (48 tests; 27 unit + 21 integration that auto-skip if server down)
cd copalvx/client && uv run pytest
```

Server changes: also run a Class A or B deploy following [copalvx/server/DEPLOY.md](./copalvx/server/DEPLOY.md) — `/copal-deploy` walks through it.

| Verify cross-package contract is consistent | `/copal-cross-package` |

## 9. When this protocol is wrong

This document is a default, not a contract. Skip steps when they don't apply (typo fix → no test, no docs, just commit). Override the loop when the user explicitly asks. Update this file when a workflow rule keeps biting and the documented rule is the cause.
